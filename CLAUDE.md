# CLAUDE.md

用于减少常见 LLM 编程错误的行为准则。可与项目专属指令合并使用。

**取舍说明：** 这些准则更偏向"谨慎"而非"速度"。对于琐碎任务，请自行判断是否有必要严格遵守。

## 1. 回答语言

- **默认使用中文回答。**
- **专业技术用语保留英文**，不强行翻译成中文（比如 API、endpoint、commit、pull request、race condition、singleton 这类词，直接用英文；其余说明性文字用中文）。
- 代码里的变量名/函数名按行业惯例用英文；代码注释默认用中文，除非项目已有的注释是英文（那就匹配现有风格，参照第 3 条"精准修改"）。
- commit message 默认中文，除非项目已有的 commit history 是英文（同样匹配现有风格）。

## 2. 先思考，再写代码

**不要瞎猜。不要隐藏困惑。主动暴露权衡点。**

实现之前：
- 明确说出你的假设。如果不确定，就问。
- 如果存在多种理解方式，把它们都列出来——不要偷偷选一个就往下做。
- 如果有更简单的方案，说出来。该反对的时候就反对。
- 如果有什么不清楚，停下来。说清楚哪里让你困惑。然后提问。

## 3. 简洁优先

**用能解决问题的最少代码。不写任何投机性的东西。**

- 不做任务之外的功能。
- 不为只用一次的代码搞抽象。
- 不加没被要求过的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行本可以是 50 行的代码，重写它。

问自己一句："一个资深工程师会不会觉得这写复杂了？"如果会，就简化。

## 4. 精准修改

**只动你必须动的地方。只清理你自己造成的混乱。**

修改已有代码时：
- 不要去"优化"旁边不相关的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有代码风格，即使你自己会写得不一样。
- 如果你发现了无关的死代码，提一下——不要顺手删掉。

当你的改动产生了孤儿代码时：
- 删除因为**你的改动**而变得没用的 import/变量/函数。
- 不要删除本来就存在的死代码，除非被明确要求。

检验标准：每一行改动都应该能直接追溯到用户的需求。

## 5. 目标驱动执行

**先定义"完成"的判定标准，再循环验证直到达标。**

把任务转化成可验证的目标：
- "加个校验" → "先写针对非法输入的测试，再让测试通过"
- "修这个 bug" → "先写一个能复现 bug 的测试，再让它通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，先说明一个简要计划：
```
1. [步骤] → 验证方式：[检查项]
2. [步骤] → 验证方式：[检查项]
3. [步骤] → 验证方式：[检查项]
```

强判定标准能让你独立循环推进；弱判定标准（比如"让它能跑起来"）则需要不断地来回确认。

## 6. 环境与依赖管理

- **统一使用 `uv` 管理 Python 环境和依赖，不使用 `pip` / `conda` / `poetry`。**
- 新项目用 `uv init` / `uv add` 建立依赖，不手写 `requirements.txt`（除非目标环境明确只认 `requirements.txt`，这种情况下用 `uv export` 生成，而不是手动维护）。
- 涉及虚拟环境操作时，默认用 `uv venv` / `uv run`，不假设用户已经 `source activate` 了某个 conda 环境。
- 调用 LLM API 所需的 HTTP 库（如 `httpx`、`openai`）以及 JSON 处理相关依赖，统一通过 `uv add` 管理。
- 如果项目里已经有 `requirements.txt` 或 `environment.yml` 之类的遗留配置，先按第 3 条"精准修改"原则处理：不要顺手把整个项目迁移成 `uv`，除非明确被要求这么做。

## 7. 命令执行提示

当需要执行可能产生副作用的命令（如文件操作、网络请求、安装依赖、运行脚本等）并请求用户许可时：
- **先用中文说明这一步要做什么。**
- **然后说明可能产生的后果**（比如会创建什么文件、会往哪里写数据、会消耗多少网络资源等）。
- 在用户确认后再执行。

## 8. Superpowers 技能的使用

- **自动触发优先**：当任务涉及头脑风暴、计划编写、代码审查、系统化调试等场景时，自动调用对应的 Superpowers 技能，无需用户手动指定。
- **手动触发支持**：用户也可以通过自然语言（如"运行 brainstorming"）或 `/技能名` 格式要求手动触发某个技能。
- **技能调用前说明**：调用 Superpowers 技能前，用中文简要说明要使用哪个技能以及为什么（如"我将使用 systematic-debugging 技能来系统地排查这个 bug"）。
- **遵循技能流程**：技能被调用后，严格遵循其 SKILL.md 中定义的步骤和检查清单。

## 9. 项目上下文：AI4SE Final Project — Coding Agent Harness

> 本项目是南京大学软件学院 AI4SE 课程的期末项目。完整要求见 `AI4SE_Final_Project_A_Coding_Agent_Harness.md` + `AI4SE_Final_Project_通用要求.md`。

### 9.1 核心公式

**Agent = LLM + Harness**。LLM 只负责"想下一步"这一行任务决策；其余一切（工具、记忆、治理、反馈、配置）都是 harness 的工程。

### 9.2 六个必须实现的维度

| 维度 | 说明 | 最低要求 |
|------|------|---------|
| **决策封装** | 组织上下文 → 调用 LLM → 解析动作 → 分发执行 → 回灌结果 → 停机判断 | 自实现主循环，不寄生现成框架 |
| **动作/工具** | agent 能执行的操作（读写文件、执行 shell、运行构建与测试） | 可运行 |
| **客观反馈信号** | 运行测试/lint/类型检查，结果回灌驱动自我修正 | 代码实现校验器/传感器，非提示词 |
| **危险动作** | 护栏拦截 + HITL 人工审批 | 代码实现 guardrail 函数，非提示词 |
| **记忆** | 跨会话持久化，按需检索而非全量载入 | 存储与检索自实现，不直接接框架 memory |
| **配置** | 声明式规则约束 agent 行为 | 可运行 |

**选择一个维度深入实现作为主要贡献，其余五个维度做可运行的最低实现。**

### 9.3 硬性约束

- **必须自己实现 agent 主循环**，不允许建在 LangChain `AgentExecutor`、AutoGen、CrewAI、LlamaIndex agent 等高层循环之上。
- **必须有一个可注入 mock 的 LLM 抽象层**，可替换为 mock LLM 运行离线测试，也可接入真实供应商。
- **机制必须是代码，不是提示词**：反馈信号 = 你编写的校验器/传感器；危险动作拦截 = 你编写的护栏。移除真实 LLM 后，每个核心机制仍能用确定性单元测试验证。
- **TDD 是硬性要求**：先红、再绿、再重构。不接受先写实现再补测试。
- **凭据绝不硬编码**：key 不进入源码、不提交 Git、不写入日志。实现安全存储（钥匙串/加密文件/密钥管理服务）。
- **实现语言**：TypeScript（使用 `npm` 管理依赖）。
- **LLM 供应商**：待定。

### 9.4 工作流程

严格遵循 Superpowers 七步工作流，顺序如下：

1. **brainstorming** → 产出 `SPEC.md`
2. **writing-plans** → 产出 `PLAN.md`
3. **冷启动验证** → 用第二个 agent 验证 SPEC + PLAN，产出 `SPEC_PROCESS.md`
4. **实现** → git worktrees + subagent 驱动 + TDD + 两阶段评审
5. **finishing-a-development-branch** → merge/PR/保留/丢弃
6. **分发与 CI** → Docker/二进制 + GitHub Actions

### 9.5 交付物清单

1. `SPEC.md` — 设计文档（含领域与机制设计一节）
2. `PLAN.md` — 实现计划
3. `SPEC_PROCESS.md` — 过程文档
4. 完整源代码（规范的 commit/PR 历史，无真实凭据）
5. 分发产物与说明（Dockerfile/构建脚本）
6. `README.md` — 项目简介、安装、运行、分发、安全边界
7. `AGENT_LOG.md` — 过程日志
8. CI 配置（`.github/workflows/`，含 `unit-test` job）
9. `REFLECTION.md` — 1500–2500 字反思报告
10. 机制演示 — mock LLM 下确定性复现护栏拦截、反馈闭环、重点维度行为

### 9.6 重要原则

- **在 SPEC 与 PLAN 完成并通过冷启动验证之前，禁止编写任何实现代码。**
- **每一行改动都应直接追溯到项目需求**——不写投机性代码，不做任务之外的功能。
- **遇到不确定之处先暂停询问**，不凭猜测继续。

---

**这些准则生效的标志：** diff 里没有多余的改动、因为过度设计导致的重写变少，以及澄清性问题出现在写代码**之前**而不是出错**之后**。
